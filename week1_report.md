### 2. 数据探索分析（EDA）
对原始数据集（10,000 条中日印尼语句对）进行全面分析，发现两个主要问题：

| 问题 | 数量 | 占比 |
|------|------|------|
| 完全重复样本 | 5,509 条 | 55.1% |
| 混语样本（译文含未翻译中文术语） | 10,000 条 | 100%（日语、印尼语均 100%） |

**混语问题说明**：日语和印尼语译文中大量保留了原始中文术语（如 `API接口调用`、`资源调度`），未进行翻译。这与项目背景中提到的"传统NLP翻译在技术术语准确性上仍有提升空间"直接吻合，也通过实验验证了通用翻译模型（Google Translate）在云计算术语上的局限性——例如将"事务"译为日常用语"urusan"、将"快照"译为"Foto（照片）"、将"镜像仓库"译为"鏡の倉庫（镜子的仓库）"。

**未翻译术语提取**：在混语计数之外，从印尼语译文中完整提取了未翻译中文术语清单——共 **35 个独立术语、累计出现 23,029 次**，高频集中在 `故障恢复`（1,712 次）、`数据存储`（1,692 次）、`安全防护`（1,678 次）等。该清单（含频率）是术语表构建和批量替换的直接输入（见 `logs/untranslated_terms_top20.png`）。

**日语混语检测方法**：日语本身使用汉字，字符占比法无法区分正常日文汉字（如 関連、障害）与未翻译中文，因此对日语改用两条互补思路：
- **术语匹配**：用从印尼语提取的术语清单（确定为未翻译残留）在日语译文中精确查找 → 日语混语率同为 **100%**，残留的是同一批术语，形态为"简体术语作为名词整体嵌在正常日语句子骨架中"（如 `ポッドをデプロイする前に、API【接口调用】と【批量处理】の準備作業を…`），因此可与印尼语共用同一套"术语表整词替换"策略，无需分词；
- **简体字形检测**：中文简化字与日语汉字字形不同（资/调/务 vs 資/調/務），维护一份"简体字→日语字形"对照，以简体特有字符作混语信号，可兜底发现术语清单之外的残留；其盲区是全部由中日同形字构成的术语（如 `快照`、`算力`），恰好由术语匹配覆盖，两法交叉验证在本数据集上结论一致（漏检 0 条）。

**重复样本分布特征**：重复并非均匀分布，明显分两档——操作/运维类 topic（对象存储使用 94.7%、容器服务部署 94.2%、数据库运维 93.4%、云计算基础 91.2%、云服务器操作 91.0%）显著高于配置/排查类（约 74%），说明操作类场景句式模板更单一、生成多样性不足；单句重复 2–8 次、平均 2.9 次，属"普遍轻度重复"而非少数句子被大量复制，指向数据生成阶段的全局性重复采样；2,858 组重复句均不跨 topic，`drop_duplicates` 不会破坏 topic 分布（见 `logs/duplicate_by_topic.png`）。

### 3. 数据清洗
**去重**：删除完全重复样本，保留 4,491 条有效数据。

**术语表构建**（`data/glossary_reviewed.json`）：
- 使用正则提取印尼语译文中残留的中文术语，共发现 35 个独立术语
- 调用 Google Translate API 自动生成初始术语对照表
- 人工审核并修正 18 处错误，主要问题类型：
  - 日常用语误用：`事务→トランザクション`、`快照→snapshot`
  - 日语助词冗余：`リソースのスケジューリング→リソーススケジューリング`
  - IT领域惯例：`带宽→bandwidth`、`节点→node`、`镜像仓库→container registry`

**混语替换**：基于审核后术语表，对日语、印尼语译文批量替换（长词优先，避免短词打断长词），替换后做三重验证：

| 验证指标 | 替换前 | 替换后 |
|------|--------|--------|
| 印尼语混语样本（字符占比法） | 4,491 条（100%） | 0 条 |
| 日语含术语表中文术语（术语匹配法） | 4,491 条（100%） | 0 条 |
| 日语含简体特有字符（字形兜底法） | 4,491 条（100%） | 0 条 |

注：字符占比法对日语无效（正常日文含汉字，会误报），故日语用"术语匹配 + 简体字形"双路验证，方法见 EDA 部分。

### 4. 数据集划分
按 topic 分层随机采样，划分比例 80/10/10：

| 数据集 | 条数 | 用途 |
|--------|------|------|
| 训练集（train） | 3,592 | 模型训练 |
| 验证集（val） | 449 | 训练过程监控、调参 |
| 测试集（test） | 450 | 最终效果评估 |

所有 topic 在三个集合中分布均匀，分层采样生效。

## 三、遗留问题与待确认事项

1. **术语表待专家审核**：当前术语表基于 Google Translate + 人工修正，部分术语（如`负载均衡配置`的日语译法）可能与阿里云官方文档标准不一致，建议由有日语/印尼语背景的专家确认。

2. **日语混语的检测边界**：已用"术语匹配 + 简体字形"双路检测，替换后日语残留验证为 0。但两种方法各有边界——术语匹配依赖已提取的术语清单，简体字形检测对纯中日同形字术语（如 `快照`、`算力`）无效。若未来新数据中出现"清单之外且全部由同形字构成"的新术语，可能同时绕过两路检测，建议每次数据更新后重跑术语提取流程，并对少量样本做人工抽检。

3. **数据规模**：清洗后有效数据 4,491 条，去重损失较大（55%），且重复集中在操作/运维类 topic（重复率 91–95%，配置/排查类约 74%），说明这类场景的生成句式模板单一。后续如需扩充训练数据，建议评估原始数据的生成方式、从源头控制重复采样，并优先补充高重复 topic 的多样化句式。
